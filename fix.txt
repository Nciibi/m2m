4. PCP

This one is important.

Your comments say

24 byte request

but the actual PCP MAP message defined in RFC 6887 is considerably larger because it includes fields such as the client IP address, a mandatory 96-bit nonce, protocol, ports, and external address. The request layout in your implementation is simplified.

For many home routers this may work if they're tolerant, but it won't be interoperable with every PCP implementation.

If you want RFC-compliant PCP, you'll eventually want to build the packet exactly as specified.

The remaining gap isn't in the TCP logic itself—it's in evolving from a single hole-punch function into a complete ICE-like connection manager.

If M2M is intended to be a flagship secure P2P messenger, that's the architecture I'd aim for. It scales naturally as you add IPv6, UPnP, STUN, TCP hole punching, relay support, and future transports without turning one module into a large collection of special cases.
