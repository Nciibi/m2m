Why do we even need STUN?

The entire reason STUN exists is because of NAT.

With IPv4:

Alice

192.168.1.10

is hidden behind

203.0.113.50

The router translates everything.

That translation is NAT.

NAT is what makes P2P difficult.

IPv6 has no NAT (normally)

Instead every device gets its own public address.

Example:

Alice

2001:db8:1234:5678::42

Bob

2001:db8:abcd:100::9

Those addresses are globally routable.

Alice can directly send packets to Bob.

Bob can directly send packets to Alice.

No translation.

What does the network look like?

IPv4

Alice
   │
Router (NAT)
   │
Internet
   │
Router (NAT)
   │
Bob

IPv6

Alice
   │
Internet
   │
Bob

Much simpler.

Does that mean no router?

No.

The router still exists.

Alice
   │
Router
   │
Internet

But the router is routing, not rewriting addresses.

Is it always open?

Not necessarily.

Many IPv6 routers have a firewall.

Example

Internet

↓

Router

↓

DROP

or

Internet

↓

Router

↓

ALLOW

So having IPv6 doesn't guarantee incoming connections are allowed, but when they are, you don't need NAT traversal.

Detecting IPv6

Rust makes this easy.

Instead of only asking for IPv4 addresses

local_ip()

iterate all interfaces.

You'll get

127.0.0.1

192.168.1.20

fe80::....

2001:db8::....

Ignore

fe80::

Those are link-local.

They only work on the local LAN.

Use

2000::/3

which are Global Unicast Addresses.

Listening

Instead of

TcpListener::bind("0.0.0.0:9000")

listen on

"[::]:9000"

or create both an IPv4 and IPv6 listener, depending on platform behavior.

On many operating systems, an IPv6 socket can also accept IPv4 connections (dual-stack), but on others you'll need two listeners.

Candidate priority

I'd make IPv6 one of the highest priorities.

Example

Candidate	Priority
LAN IPv4	100
IPv6 Global	95

Because if both peers have working global IPv6, it's a direct connection with no NAT traversal overhead.

The remaining gap isn't in the TCP logic itself—it's in evolving from a single hole-punch function into a complete ICE-like connection manager.

If M2M is intended to be a flagship secure P2P messenger, that's the architecture I'd aim for. It scales naturally as you add IPv6, UPnP, STUN, TCP hole punching, relay support, and future transports without turning one module into a large collection of special cases.
