# Mission: Implement Phase 3 — Secure Group Chat

You are implementing **Phase 3 (Group Chat)** of this secure messaging application.

This is a **security-critical feature**. The implementation must meet the quality standards expected of a production secure messenger.

Your priorities, in order, are:

1. Cryptographic correctness
2. Protocol correctness
3. Security
4. Reliability
5. Maintainability
6. Performance

Never sacrifice correctness for speed.

---

# Existing Architecture

This project already contains:

* X3DH key agreement
* Double Ratchet sessions
* Ed25519 identities
* Secure encrypted storage
* Packet-based protocol
* Rust backend
* React/Tauri frontend

The new Group Chat implementation **must integrate with the existing architecture** instead of bypassing or replacing it.

Do not create parallel systems when existing infrastructure can be reused.

# Cryptographic Requirements

Treat all cryptographic operations as security-critical.

Never invent cryptographic behavior.

Use only the project's existing cryptographic primitives together with the approved Sender Key design.

Ensure:

* correct HKDF usage
* correct chain-key advancement
* unique message keys
* replay protection
* signature verification
* authenticated sender identity
* proper key rotation
* secure member removal
* secure key distribution
* encrypted storage of sensitive material

Zero plaintext keys should ever be persisted.

Never log secrets.

Never expose private material through debugging.

---

# Protocol Requirements

The protocol implementation must be deterministic and robust.

Handle:

* duplicate packets
* missing packets
* reordered packets
* invalid signatures
* invalid Sender Keys
* malformed packets
* unknown members
* removed members
* stale Sender Keys
* packet version mismatches

Gracefully reject invalid input without crashing.
